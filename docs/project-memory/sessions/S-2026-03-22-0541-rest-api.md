## Goal

Build a FastAPI REST API that exposes the Sprint 1 embedding store via HTTP endpoints for search, clustering, and repo detail.

## Context

Sprint 1 built the embedding store (VectorStore, EmbeddingPipeline). Sprint 2 agentB is responsible for wrapping this in a REST API.

## Plan

1. Add fastapi, uvicorn, scikit-learn to pyproject.toml
2. Create clusters.py with KMeans-based ClusterEngine
3. Create api.py with FastAPI app: /api/search, /api/clusters, /api/repos/<slug>, /api/health
4. Add ghps-server console script entry
5. Verify tests pass and server starts

## Changes Made

- **pyproject.toml**: Added fastapi, uvicorn, scikit-learn deps; added `[project.scripts]` ghps-server entry
- **src/ghps/api.py**: FastAPI app with 4 endpoints, CORS middleware, lifespan startup/shutdown, CLI with --port/--db flags, stale server cleanup
- **src/ghps/clusters.py**: ClusterEngine using KMeans on per-repo average embeddings, auto-generated cluster names from language/topics metadata

## Decisions Made

- Used distance-to-score conversion (1.0 - distance) since VectorStore returns distance (lower=better) but API consumers expect score (higher=better)
- Auto-generate cluster names from most common language + topic in each cluster
- CORS allows all origins for simplicity in dev/portfolio context
- Consistent JSON envelope: {"ok": true, "data": ...} / {"ok": false, "error": "..."}

## Open Questions

- SearchEngine in search.py has bugs (references row["score"] and row["url"] that don't exist) — API works around this by calling store.search() directly
- Cluster names could be improved with LLM-generated labels in a future sprint
