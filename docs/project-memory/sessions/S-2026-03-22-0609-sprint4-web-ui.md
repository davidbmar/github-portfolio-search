## Goal

Build the public web UI for davidbmar.com with search and browse capabilities as a vanilla JS single-page application served from static files.

## Context

Sprint 4 task. The backend (FastAPI + SQLite-vec) and data pipeline exist. This sprint adds the frontend that loads pre-exported repos.json and clusters.json for client-side search and browsing.

## Plan

1. Create web/css/style.css — dark theme
2. Create web/js/search.js — client-side search engine
3. Create web/js/app.js — SPA with hash routing, rendering, filters
4. Create web/index.html — entry point

## Changes Made

- **web/css/style.css**: Dark theme with CSS custom properties. Mobile-responsive with breakpoints at 768px and 375px. Collapsible filter sidebar on mobile.
- **web/js/search.js**: Client-side search engine with weighted scoring (name 10pt, topics 6pt, description 3pt, README 1pt). AND semantics across terms. Facet extraction and filtering by language, topics, stars.
- **web/js/app.js**: Main SPA module. Hash routing (#/, #/search?q=X, #/cluster/name). Debounced search input (300ms). Loads repos.json + clusters.json on init. XSS-safe rendering via escapeHtml/escapeAttr.
- **web/index.html**: Single-page app shell with header search bar and content area.

## Decisions Made

- **Vanilla JS with IIFE modules**: No build step needed, fits S3 static hosting constraint. SearchEngine and App are separate IIFE modules.
- **Client-side search over API calls**: All data loaded upfront for instant search after initial load. Trade-off: heavier initial load but zero latency per keystroke.
- **Hash routing**: Works with S3 static hosting without server-side rewrites.
- **Weighted scoring**: Name matches weighted highest (developers typically search by name), then topics, description, README.
